Permission Slip API (1.0.0)

Download OpenAPI specification:

URL: https://github.com/supersuit-tech/permission-slip-web

Permission Slip is a centralized middle-man service that sits between AI agents and external APIs. Agents submit pre-defined actions for human approval, and Permission Slip executes them using the user's stored credentials. Agents never see user credentials and can never make arbitrary API calls.

Base URL: https://app.permissionslip.dev

Key Concepts

  • Action: Pre-defined, structured request with a type, validated parameters, and execution logic — the core primitive
  • Agent: Automated system with a cryptographic key pair that submits actions for approval
  • Approver: Human authorized to approve or deny agent actions
  • Connector: Integration with an external service (e.g., Gmail, Stripe) that defines available actions
  • Credential: Encrypted API key or OAuth token for an external service, stored in Permission Slip's vault
  • Registration: One-time setup where agent proves identity via public key
  • Approval Request: Async request for permission to execute a specific action
  • Token: Single-use credential issued after one-off approval, used to execute the approved action
  • Standing Approval: Pre-authorized, constraint-scoped grant that lets an agent execute an action repeatedly without per-request approval

Authentication

Permission Slip uses two authentication schemes, plus some public discovery endpoints:

  • Agent-facing endpoints (registration, approvals, actions) require Ed25519 request signatures. See the PermissionSlipSignature security scheme.
  • User-facing endpoints (agents, credentials, standing approvals, profiles) use Supabase session JWTs. See the SupabaseSession security scheme.
  • Discovery endpoints (connectors) are publicly accessible with no authentication required.

Workflow

One-off Approval (default)

  1. Discover Actions: Agent queries GET /v1/connectors to see available actions
  2. Registration: Agent registers by providing public key, user approves
  3. Store Credentials (if needed): User connects external accounts and stores credentials in Permission Slip's vault using the credential management endpoints
  4. Submit Action: Agent submits a structured action (e.g., email.send with parameters)
  5. User Approval: Approver reviews the action in a human-readable format and approves/denies
  6. Verify & Get Token: Agent submits confirmation code, receives single-use token
  7. Execute Action: Agent calls POST /v1/actions/execute with the token — Permission Slip calls the external API using stored credentials

Standing Approval (pre-authorized)

  1. User creates standing approval: Via web UI — specifies agent, action type, constraints, duration, and optional execution cap
  2. Agent executes: Calls POST /v1/actions/execute with agent signature (no token) — Permission Slip matches the standing approval and executes immediately
  3. Fallthrough: If no standing approval matches, falls through to the one-off flow above

Architecture

Agent ──→ Permission Slip ──→ External Service (Gmail, Stripe, etc.)
               │
               ▼
          User (approve/deny)

Permission Slip is the single integration point for agents. Agents talk to https://app.permissionslip.dev for everything — they never need to know about individual service URLs or APIs.

Registration

Agent registration and verification

Register Agent via Invite URL

Register a new agent with Permission Slip by POSTing to the invite URL. The invite code in the URL path identifies which user the agent is registering with — the agent does not need to know the user's username.

The invite URL is generated by the user from the Permission Slip dashboard (POST /v1/registration-invites) and shared with the agent. The agent POSTs its public key to the URL to begin registration.

Registration Flow

  1. User generates invite: Clicks "Add Agent" on dashboard, receives invite URL (e.g., https://app.permissionslip.dev/invite/PS-R7K3-X9M4)
  2. User shares URL: Gives the invite URL to the agent (paste in chat, config file, environment variable, etc.)
  3. Agent POSTs to invite URL: Sends public key and request ID
  4. Permission Slip validates: Verifies the invite code is valid, not expired, and not consumed — then creates a pending registration
  5. Dashboard shows confirmation code: User sees a 6-character confirmation code on their dashboard
  6. User shares confirmation code: Communicates the code to the agent (out-of-band)
  7. Agent verifies: Submits the code to POST /v1/agents/{agent_id}/verify
  8. Registration complete: Agent can now make approval requests

Why no username?

The invite code already maps to exactly one user in the database. When the user generates an invite, the backend stores their user_id alongside the hashed invite code. When the agent POSTs to the invite URL, the backend looks up the invite code and finds the user automatically. Requiring the agent to also specify the username would be redundant.

Security

  • Request MUST include valid X-Permission-Slip-Signature header
  • The server auto-assigns an agent_id (bigint) returned in the response
  • Permission Slip enforces 5-minute timestamp window for signatures
  • Duplicate request_id values are rejected within timestamp window
  • Invite codes are single-use, time-limited (default 15 min), and locked after 5 failed attempts

Idempotency

If the same request_id is used multiple times (within 300 seconds), Permission Slip returns the original response if registration succeeded, or rejects with duplicate_request_id error.

Verify Agent Registration

Complete agent registration by submitting the confirmation code shown to the user on the Permission Slip dashboard.

Verification Flow

  1. Agent registers: POSTs to invite URL (POST /invite/{invite_code})
  2. Code displayed: Permission Slip shows 6-character confirmation code on the user's dashboard (e.g., XK7-M9P)
  3. User provides code: User communicates code to agent (out-of-band)
  4. Agent submits code: Calls this endpoint with code
  5. Service verifies: Checks code matches, hasn't expired, attempts < 5
  6. Registration complete: Agent is now registered and can make approval requests

Confirmation Code Format

  • Length: 6 characters (may include hyphen for readability: XXX-XXX)
  • Character set: Uppercase alphanumeric excluding confusing chars (0, O, 1, I)
  • Allowed: ABCDEFGHJKLMNPQRSTUVWXYZ23456789 (32 characters)
  • Case-insensitive: Permission Slip normalizes to uppercase before validation
  • Examples: XK7-M9P, XK7M9P, xk7-m9p (all equivalent)

Security

  • Attempt limit: 5 incorrect attempts allowed
  • Lockout: After 5 failures, registration expires (must re-register)
  • Timing: Constant-time comparison to prevent timing attacks
  • Single-use: Code invalidated after successful verification
  • Expiration: Code expires with registration (typically 5 minutes)

Path Parameter Validation

The agent_id in the URL path MUST match the agent_id in the X-Permission-Slip-Signature header. If mismatch, Permission Slip returns 400 Bad Request with error code agent_id_mismatch.

Get Agent Capabilities

Returns everything an agent needs to know about what it can do on behalf of a specific user — enabled connectors, available actions with full parameter schemas, active standing approvals, and credential readiness.

This is the primary discovery endpoint for agents. Call it at startup (or periodically) to build a complete picture of available capabilities before submitting approval requests or executing actions.

What's Included

For each connector enabled for this agent:

  • Connector metadata: ID, name, description
  • Credential readiness: Whether the user has stored the required credentials (boolean only — no credential values are ever exposed)
  • Actions: Full list of available action types with:
    • Human-readable name and description
    • Risk level (low, medium, high)
    • Parameter JSON Schema (for client-side validation)
    • Active standing approvals (constraints, remaining executions, expiration)

What's NOT Included

  • Connectors that are not enabled for this agent (use GET /v1/connectors for the full global catalog)
  • Deactivated or expired standing approvals
  • Credential values, types, or metadata (only presence/absence)

Agent Workflow

After calling this endpoint, the agent knows:

  1. Which connectors are available → connectors[]
  2. Which are ready to use → credentials_ready: true
  3. Which need user setup → credentials_ready: false + credentials_setup_url
  4. What actions exist → connectors[].actions[]
  5. What parameters each action expects → parameters_schema
  6. Which actions can execute immediately → standing_approvals is non-empty
  7. Which need one-off approval → standing_approvals is empty
  8. What constraints apply → standing_approvals[].constraints

Authorization Flow Decision

For a given action the agent wants to perform:

Has standing approval with matching constraints?
  ├─ YES → POST /v1/actions/execute (no token, agent signature only)
  └─ NO  → POST /v1/approvals/request → user approves → verify → execute with token

If credentials_ready is false for the connector, the agent should prompt the user to visit credentials_setup_url before attempting any actions.

Security

  • Request MUST include valid X-Permission-Slip-Signature header
  • The agent_id in the URL path MUST match the agent_id in the signature header
  • Only connectors and standing approvals belonging to the authenticated agent's user are returned
  • Credential values are never exposed — only a boolean readiness flag

Approvals

Approval request lifecycle management

Request Approval for Action

Request approval for a specific one-off action. The designated approver must review and approve the action before the agent can execute it.

Approval Request Flow

  1. Agent requests: Sends action details, parameters, and context
  2. Permission Slip validates: Verifies agent is registered, signature is valid
  3. Permission Slip responds: Returns approval URL with unique approval_id
  4. Notification sent: Permission Slip notifies approver (push notification)
  5. Approver reviews: Visits URL, sees full action details, approves/denies
  6. Confirmation shown: If approved, Permission Slip shows 6-character confirmation code
  7. Agent verifies: Collects code from user, submits to verify endpoint
  8. Token issued: Permission Slip returns single-use JWT token
  9. Agent executes: Uses token to call POST /v1/actions/execute

Action Types

Action types follow naming conventions:

Standard Types (2 segments, reserved for future standardization):

  • Format: <category>.<operation> (e.g., email.send, payment.charge)
  • Pattern: ^[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*$
  • Max length: 64 characters

Custom Types (4+ segments, service-specific):

  • Format: <reverse-domain>.<category>.<operation> (e.g., com.example.deploy.production)
  • Pattern: Reverse-DNS domain (2+ segments) + category + operation
  • Max length: 128 characters

Permission Slip validates action types and returns 400 Bad Request for invalid formats or unsupported types.

Security

  • Agent MUST be registered with this approver
  • Request MUST include valid X-Permission-Slip-Signature header
  • The agent_id in request body MUST match signature header
  • Duplicate request_id rejected within timestamp window
  • Approval expires after TTL (typically 5 minutes)

Parameters Hash

The action.parameters object is canonicalized using RFC 8785 (JCS) and hashed with SHA-256. This hash is embedded in the approval token as params_hash to prevent parameter tampering. Permission Slip verifies the hash matches request parameters before executing actions.

Verify Approval and Retrieve Token

Verify approval and retrieve the single-use token by submitting the confirmation code shown to the approver after they approved the action.

Verification Flow

  1. Approver approves: Visits approval URL, reviews action details, approves
  2. Code displayed: Permission Slip shows 6-character confirmation code (e.g., RK3-P7M)
  3. User provides code: Approver communicates code to agent owner/operator
  4. Agent submits code: Calls this endpoint with code
  5. Permission Slip verifies: Checks code matches, hasn't expired, attempts < 5
  6. Token issued: Permission Slip returns single-use JWT token with action scope
  7. Agent uses token: Passes token to POST /v1/actions/execute to trigger execution

Denial Handling

If the approver explicitly denied the approval request, this endpoint returns 403 Forbidden with error code approval_denied. The agent should NOT retry or re-request approval for the same action without user intervention.

Token Structure

The returned JWT token contains:

  • sub: Agent ID
  • aud: permissionslip.dev
  • approver: Username who approved
  • approval_id: Approval request ID
  • scope: Action type (e.g., email.send)
  • scope_version: Action version (e.g., "1")
  • params_hash: SHA-256 hash of JCS-canonicalized action parameters
  • iat, exp: Issued/expires timestamps
  • jti: Unique token ID (for single-use enforcement)

Token Usage

The token is passed to POST /v1/actions/execute along with the action ID and parameters. Permission Slip verifies the token and executes the action using the user's stored credentials:

POST https://app.permissionslip.dev/v1/actions/execute
X-Permission-Slip-Signature: agent_id="...", algorithm="Ed25519", ...
Content-Type: application/json

{
  "token": "<approval_token>",
  "action_id": "email.send",
  "parameters": { ... }
}

Permission Slip verifies:

  1. JWT signature (token was issued by Permission Slip)
  2. Expiration (exp claim)
  3. Scope matches action_id
  4. Parameters hash matches request parameters
  5. Token not already used (jti tracking)

Security

  • Attempt limit: 5 incorrect code attempts allowed
  • Lockout: After 5 failures, approval expires
  • Single-use code: Code invalidated after successful verification
  • Single-use token: Token invalidated after first successful action
  • Expiration: Both code and token expire with approval TTL

Actions

Execute approved actions via Permission Slip

Execute Action

Execute an action via Permission Slip. This endpoint supports two authorization modes:

Mode 1: One-off Approval (Bearer Token)

The agent redeems a single-use approval token from the one-off approval flow.

  1. Agent submits token: Provides the approval token, action ID, and parameters
  2. Permission Slip validates: Verifies token signature, expiration, scope, and parameter hash
  3. Permission Slip executes: Calls the external service API using the user's stored credentials
  4. Result returned: External service response is returned to the agent
  5. Token consumed: Token is invalidated (single-use)

Mode 2: Standing Approval (Agent Signature)

The agent executes under a pre-authorized standing approval — no token needed.

  1. Agent submits action: Provides action type and parameters (no token)
  2. Permission Slip matches: Finds an active standing approval for this agent + action + constraints
  3. Permission Slip validates: Verifies parameters fall within standing approval constraints
  4. Permission Slip executes: Calls the external service API immediately
  5. Result returned: Uses the StandingApprovalExecuteResponse schema (includes standing_approval_id and executions_remaining)

Permission Slip determines the mode by the presence of the token field in the JSON request body. If a non-empty token is provided, one-off token verification is used. If token is absent, standing approval matching is attempted using the agent signature. If no standing approval matches, the request falls through to the one-off approval flow — Permission Slip returns 404 Not Found with a hint to use POST /v1/approvals/request, indicating the agent should initiate the standard approval flow for this action.

Token Validation (One-off mode)

Permission Slip verifies:

  • JWT signature: Token was issued by Permission Slip
  • Expiration: exp claim has not passed
  • Scope: scope claim matches the action_id in the request
  • Parameters hash: SHA-256 of JCS-canonicalized parameters matches params_hash claim
  • Single-use: Token jti has not been previously consumed

Standing Approval Validation (Standing approval mode)

Permission Slip verifies:

  • Agent signature: Valid Ed25519/ECDSA signature
  • Standing approval match: Active standing approval for this agent + action type
  • Constraint compliance: Parameters fall within standing approval constraints
  • Expiration: Standing approval has not expired (or has no expiration)
  • Execution count: Max executions not exceeded (if set)

Credential Lookup

Permission Slip determines which external service to call based on the action type (via connector configuration) and retrieves the user's stored credentials from the encrypted vault. The agent never sees these credentials.

Error Handling

If the external service returns an error, Permission Slip returns a 502 Bad Gateway with details about the upstream failure. For one-off mode, the token is still consumed — the agent must request a new approval to retry. For standing approval mode, the execution count is not incremented on upstream failures.

Security

  • Request MUST include valid X-Permission-Slip-Signature header
  • One-off: Token is single-use — consumed even if the external service call fails
  • Standing: Constraints are enforced on every execution
  • Parameters must match what was approved (hash verification for one-off, constraint check for standing)
  • Agent never sees user credentials

Credentials

Manage stored credentials for external services

Store Credentials

Store API credentials for an external service in Permission Slip's encrypted vault. These credentials are used when executing approved actions — Permission Slip calls external APIs on behalf of the user using these stored credentials.

This is a user-facing endpoint — users manage credentials through the dashboard, not through agents. Agents never see or interact with credentials.

Credential Storage

  • Credentials are encrypted at rest using AES-256-GCM
  • Only decrypted at the moment of action execution
  • Never exposed to agents or included in API responses
  • Users can store multiple credential sets for the same service (e.g., work and personal GitHub tokens)
  • Credentials are user-scoped: the same credential is shared across all of the user's agents

Setup Flow

  1. User checks which credentials are needed: GET /v1/connectors/{id}
  2. User stores credentials: POST /v1/credentials
  3. Credentials are encrypted and stored in the vault
  4. Actions requiring these credentials can now be executed

Security

  • Request MUST include valid Supabase session JWT (Authorization: Bearer )
  • Credentials are encrypted immediately upon receipt
  • Only the user who stored the credentials can use or delete them

Standing Approvals

Manage standing approvals for pre-authorized agent actions

List Agent Standing Approvals

List active standing approvals for a specific agent. Standing approvals are pre-authorized, constraint-scoped grants that let an agent execute a specific action type without per-request human approval.

Agents use this endpoint to discover which actions they can execute immediately (via standing approval) vs. which require the one-off approval flow.

Use Cases

  • Agent startup: Agent queries its standing approvals to understand what it can do
  • Flow selection: Agent checks if a standing approval covers a desired action before deciding whether to use POST /v1/actions/execute (standing) or POST /v1/approvals/request (one-off)
  • Status monitoring: Agent checks remaining executions or expiration times

Filtering

Only standing approvals with status active are returned by default. Expired, revoked, and exhausted standing approvals are excluded.

Security

  • Request MUST include valid X-Permission-Slip-Signature header
  • The agent_id in the URL path MUST match the agent_id in the signature header
  • Only standing approvals belonging to the authenticated agent are returned
