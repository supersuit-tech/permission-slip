# Database Schema

The database uses PostgreSQL (via Supabase in production, standalone Postgres in development). The Go backend connects using pgx; the frontend never talks to the database directly.

Migrations are managed by [goose](https://github.com/pressly/goose) and live in `db/migrations/`. They are embedded into the Go binary and run automatically on startup.

### `registration_invites`

Single-use, time-limited invite codes generated by users to authorize agent registration. The user creates an invite from the dashboard and communicates the code to the agent out-of-band. See [ADR-005](adr/005-user-initiated-registration.md).

| Column | Type | Constraints |
|---|---|---|
| `id` | text | PK, max 255 chars |
| `user_id` | uuid | NOT NULL, FK → profiles(id) ON DELETE CASCADE |
| `invite_code_hash` | text | NOT NULL, HMAC-SHA256 or SHA-256 hash, max 128 chars |
| `status` | text | NOT NULL, CHECK IN ('active', 'consumed', 'expired') |
| `verification_attempts` | int | NOT NULL, DEFAULT 0, CHECK >= 0 |
| `expires_at` | timestamptz | NOT NULL |
| `consumed_at` | timestamptz | |
| `created_at` | timestamptz | NOT NULL, DEFAULT now() |

**Indexes:** `idx_registration_invites_user_status` on `(user_id, status)`

**Behavior:**
- Invite codes are hashed with HMAC-SHA256 when `INVITE_HMAC_KEY` is configured (recommended for production), falling back to plain SHA-256 when the key is not set. HMAC prevents hash reversal from database read access alone. Compared using constant-time comparison in the Go backend.
- Brute-force lockout at 5 failed attempts (same atomic check-and-increment pattern as agents/approvals)
- Default TTL: 15 minutes, configurable up to 24 hours
- Single-use: status transitions from `active` → `consumed` when an agent registers with the code
- Expired invites are cleaned up by pg_cron

### `agents`

Agents registered by approvers. Identified by auto-incrementing bigserial IDs. Registration uses plaintext confirmation codes with brute-force lockout.

| Column | Type | Constraints |
|---|---|---|
| `agent_id` | bigserial | PK (auto-incrementing, globally unique) |
| `public_key` | text | NOT NULL, max 1024 chars |
| `approver_id` | uuid | NOT NULL, FK → profiles(id) ON DELETE CASCADE |
| `status` | text | NOT NULL, CHECK IN ('pending', 'registered', 'deactivated') |
| `metadata` | jsonb | max 64 KB |
| `confirmation_code` | text | Plaintext code for dashboard display, format `XXX-XXX` (7 chars, DB allows up to 10) |
| `verification_attempts` | int | NOT NULL, DEFAULT 0, CHECK >= 0 |
| `registration_ttl` | int | CHECK BETWEEN 60 AND 86400 (seconds) |
| `expires_at` | timestamptz | |
| `registered_at` | timestamptz | |
| `deactivated_at` | timestamptz | |
| `last_active_at` | timestamptz | |
| `created_at` | timestamptz | NOT NULL, DEFAULT now() |

**Indexes:** `idx_agents_approver_status` on `(approver_id, status)`, `idx_agents_approver_created` on `(approver_id, created_at DESC, agent_id DESC)`

**Pagination:** The `GET /agents` endpoint uses cursor-based pagination ordered by `(created_at DESC, agent_id DESC)`. The cursor is a compound key of both fields to guarantee stable ordering even when multiple agents share the same `created_at` timestamp. The `idx_agents_approver_created` index covers this query.

**Note:** `agent_id` was migrated from a text composite PK (`agent_id, approver_id`) to a globally unique bigserial PK. All dependent tables now reference `agents(agent_id)` with a single-column FK instead of the previous composite FK.

### `credentials`

User credentials stored via Supabase Vault. The table holds a reference to the Vault secret, not the encrypted bytes directly. Credentials are **user-scoped, not agent-scoped** — the user stores their Gmail API key once and any of their agents with Gmail enabled can trigger actions that use it. Credential management is done through the dashboard using Supabase session JWT auth; agents never see or interact with credentials.

| Column | Type | Constraints |
|---|---|---|
| `id` | text | PK, max 255 chars |
| `user_id` | uuid | NOT NULL, FK → profiles(id) ON DELETE CASCADE |
| `service` | text | NOT NULL, max 255 chars |
| `label` | text | max 255 chars |
| `vault_secret_id` | uuid | NOT NULL |
| `created_at` | timestamptz | NOT NULL, DEFAULT now() |

**Constraints:** UNIQUE NULLS NOT DISTINCT on `(user_id, service, label)` — prevents duplicate credentials including when label is NULL.

**Indexes:** `idx_credentials_user_service` on `(user_id, service)`

### `agent_connectors`

Join table controlling which connectors are enabled for each agent. Users enable connectors per-agent through the dashboard — an agent can only submit actions from connectors in this table.

| Column | Type | Constraints |
|---|---|---|
| `agent_id` | bigint | PK (composite), FK → agents(agent_id) ON DELETE CASCADE |
| `approver_id` | uuid | NOT NULL |
| `connector_id` | text | PK (composite), FK → connectors(id) ON DELETE CASCADE, max 255 chars |
| `enabled_at` | timestamptz | NOT NULL, DEFAULT now() |

**Indexes:** `idx_agent_connectors_connector` on `(connector_id)`

**Behavior:**
- Enabling a connector is idempotent (ON CONFLICT DO NOTHING)
- Deleting an agent cascades to remove all its connector enablements
- Deleting a connector cascades to remove it from all agents
- This is the connector-level gating layer; a future `agent_service_actions` table may provide more granular action-level gating

### `action_configurations`

User-created configurations that define exactly how an agent is allowed to use a connector action. The configuration IS the permission — agents select from pre-configured options rather than composing requests from scratch. Parameters can be locked to specific values or set to `"*"` (wildcard) to give the agent freedom on that parameter.

| Column | Type | Constraints |
|---|---|---|
| `id` | text | PK, max 255 chars |
| `agent_id` | bigint | NOT NULL, FK → agents(agent_id) ON DELETE CASCADE |
| `user_id` | uuid | NOT NULL, FK → profiles(id) ON DELETE CASCADE |
| `connector_id` | text | NOT NULL, FK → connectors(id) ON DELETE CASCADE, max 255 chars |
| `action_type` | text | NOT NULL, max 255 chars, FK (composite with connector_id) → connector_actions ON DELETE CASCADE |
| `credential_id` | text | FK → credentials(id) ON DELETE SET NULL, max 255 chars |
| `parameters` | jsonb | NOT NULL, DEFAULT '{}', max 64 KB |
| `status` | text | NOT NULL, DEFAULT 'active', CHECK IN ('active', 'disabled') |
| `name` | text | NOT NULL, max 255 chars |
| `description` | text | max 4096 chars |
| `created_at` | timestamptz | NOT NULL, DEFAULT now() |
| `updated_at` | timestamptz | NOT NULL, DEFAULT now() |

**Indexes:** `idx_action_configurations_agent` on `(agent_id, user_id)`, `idx_action_configurations_connector_action` on `(connector_id, action_type)`

**Behavior:**
- An agent can have multiple configurations for the same action type (e.g., `email.send` to alice vs `email.send` to bob)
- Credential binding is optional — `credential_id` can be NULL (must be set before execution)
- Deleting a credential sets `credential_id` to NULL (ON DELETE SET NULL) rather than cascading the delete
- Parameters use `"*"` as a wildcard convention: fixed parameters must match exactly at execution time, wildcard parameters accept any value that passes the action's schema validation

## Relationships

Most foreign keys use ON DELETE CASCADE. Audit events use ON DELETE RESTRICT to preserve history.

```
auth.users
  └── profiles
        ├── registration_invites (user_id → profiles, CASCADE)
        ├── subscriptions (user_id → profiles, CASCADE)
        ├── usage_periods (user_id → profiles, CASCADE)
        ├── agents (approver_id → profiles, CASCADE)
        ├── approvals (approver_id → profiles, CASCADE)
        ├── credentials (user_id → profiles, CASCADE)
        ├── standing_approvals (user_id → profiles, CASCADE)
        └── audit_events (user_id → profiles, RESTRICT)

agents (PK: agent_id, bigserial)
  ├── approvals (agent_id → agents, CASCADE)
  ├── request_ids (agent_id → agents, CASCADE)
  ├── standing_approvals (agent_id → agents, CASCADE)
  ├── agent_connectors (agent_id → agents, CASCADE)

standing_approvals
  └── standing_approval_executions (standing_approval_id → standing_approvals, CASCADE)

connectors
  ├── connector_actions (connector_id → connectors, CASCADE)

```

## Conventions

### Text column length constraints

All TEXT primary keys and unbounded text columns have inline CHECK length constraints to prevent storage/index abuse from oversized values. Constraints are defined directly in the CREATE TABLE statement for each migration (no separate constraint migration). Typical limits: 255 chars for identifiers, 1024 for public keys, 128 for hashes.

### JSONB size constraints

All JSONB columns have `CHECK (pg_column_size(column) <= 65536)` constraints to prevent oversized payloads from bloating storage and indexes. The 64 KB limit applies to: `agents.metadata`, `approvals.action`, `approvals.context`, `connector_actions.parameters_schema`, and `standing_approvals.constraints`. `pg_column_size` measures the actual on-disk JSONB representation, which is more accurate than casting to text.

### pg_cron helper functions

The profiles migration defines `try_cron_schedule()` and `try_cron_unschedule()` — wrappers around `cron.schedule()` and `cron.unschedule()` that gracefully skip when pg_cron is not available (e.g. local development). All pg_cron jobs use these helpers instead of duplicating error-handling boilerplate.

### Verification attempts (brute-force lockout)

The `verification_attempts` column on `agents` and `approvals` provides brute-force lockout at 5 max attempts. The security guarantee requires an **atomic check-and-increment** in the Go backend:

```sql
-- Atomic: increment AND check in a single statement
UPDATE agents
SET verification_attempts = verification_attempts + 1
WHERE agent_id = $1
  AND verification_attempts < 5
RETURNING confirmation_code, verification_attempts;
```

If zero rows are returned, the entity is locked out. Do **not** SELECT the count first and then UPDATE separately (TOCTOU race).

The `CHECK (verification_attempts >= 0)` constraint prevents a backend bug from granting extra attempts via negative values.

### Confirmation codes

**Agents:** Stored as plaintext in `confirmation_code` (displayed on dashboard, compared using constant-time comparison). Short-lived (5-minute TTL) and low-entropy (6 chars) — hashing adds no meaningful security since the code must be displayed to the user anyway.

**Approvals:** SHA-256 hash stored in `confirmation_code_hash`, compared using constant-time comparison in the Go backend.

### Extensions

- `uuid-ossp` and `pgcrypto` — enabled in `00001_init.sql`
- `pg_cron` — enabled with graceful fallback in development (not available outside Supabase)

## Entity/Repository Layer

Query functions live in `db/` — one file per domain (e.g., `db/profiles.go`, `db/agents.go`, `db/audit_events.go`). Each function accepts a `context.Context` and `DBTX` interface (satisfied by `*pgxpool.Pool` or `pgx.Tx`), and returns typed structs without json tags (the API layer defines its own response types).

The established pattern (`db/profiles.go`):

```go
func GetProfileByUserID(ctx context.Context, pool *pgxpool.Pool, userID string) (*Profile, error) {
    var p Profile
    err := pool.QueryRow(ctx,
        "SELECT id, username, created_at FROM profiles WHERE id = $1",
        userID,
    ).Scan(&p.ID, &p.Username, &p.CreatedAt)
    if errors.Is(err, pgx.ErrNoRows) {
        return nil, nil  // not found → nil, no error
    }
    if err != nil {
        return nil, err
    }
    return &p, nil
}
```

Conventions:
- **Not-found returns `(nil, nil)`** — callers check for `nil` to distinguish "not found" from errors.
- **No json tags on DB structs** — API response types with json tags are defined in the `api/` package.
- **Handlers stay thin** — validate input, call repository functions, format responses. No raw SQL in API handlers.

### Test file organization

Tests are split by domain. Each domain file owns its own schema assertions, CASCADE tests, and constraint tests:

- `db_test.go` — foundational: connectivity, extensions
- `profiles_test.go` — schema, unique constraints, cascade from auth.users
- `connectors_test.go` — schema (connectors, connector_actions, connector_required_credentials), cascade deletes
- `agents_test.go` — schema, cascades, CHECK constraints, indexes
- `approvals_test.go` — schema, cascades, CHECK constraints, indexes, consumed tokens, pg_cron jobs
- `credentials_test.go` — schema, cascades, unique constraints (including NULL label), indexes
- `agent_connectors_test.go` — schema, cascades, indexes
- `standing_approvals_test.go` — schema, cascades, CHECK constraints (status, 90-day max, execution_count, max_executions), indexes, standing_approval_executions table
- `action_configurations_test.go` — schema, cascades, CHECK constraints (status, parameters size), credential SET NULL behavior, CRUD function tests (create, get, list, update, delete), user scoping
- `audit_events_test.go` — ListAuditEvents query: all event types, filters (agent_id, event_type, outcome), pagination, user isolation, ordering
- `subscriptions_test.go` — schema, CRUD, UNIQUE constraints
- `subscriptions_billing_test.go` — EnsureAllUsersSubscribed with billing enabled/disabled
- `subscriptions_stripe_test.go` — Stripe ID lookups (GetSubscriptionByStripeCustomerID, GetSubscriptionByStripeSubscriptionID)
- `usage_periods_test.go` — schema, atomic increments, JSONB breakdown, cascade delete
- `plans_test.go` — plan lookup, seed data verification
